# API Consistency Audit — `@strait/ui`

**Status:** Findings only — no code changes. For team review.
**Date:** 2026-05-30
**Scope:** All 124 public components in `packages/ui/src/components`.
**Author:** Engineering (design-system)

---

## Executive summary

The library's *visual and structural* contract is in excellent shape. Focus
rings, semantic colour tokens, `data-slot` tagging, `cn()` merging, `"use
client"` placement, and prop **typing style** are all machine-enforced by
[`scripts/check-conventions.mjs`](../scripts/check-conventions.mjs) and currently
pass with zero violations. Polymorphism is *almost* clean: 96 components use Base
UI's `render` prop as the contract requires, with only **2 holdouts still using
`asChild`** (`credenza.tsx`, `tree.tsx`) — a small, contained fix.

The drift lives one layer up, in the **naming and shape of the public API** —
exactly the surface a linter doesn't see and a consumer feels most. Four
classes of inconsistency stand out:

| # | Finding | Severity | Components affected |
| - | ------- | -------- | ------------------- |
| 1 | Fragmented `size` vocabulary (8 distinct scales, inconsistent ordering) | **High** | 29 |
| 2 | Semantic axis named both `variant` and `intent`; `intent` overloaded across 3 unrelated concepts | **High** | 24 |
| 3 | Boolean prop naming drift (`is*` prefix, mixed `show*`/`hide*` polarity) | **Medium** | ~30 |
| 4 | `asChild` holdouts instead of `render` prop | **Medium** | 2 |
| 5 | `*Props` type not exported by every component | **Low** | 27 |

None of these are bugs. They are **predictability debt**: each one forces a
consumer to check the docs for a component they should already be able to guess
from having used a sibling. For a design system — whose entire value
proposition is "learn it once, apply it everywhere" — predictability *is* the
product.

The team has decided on a **full rename in one breaking release** (see
[§8](#8-recommended-approach)): converge size scale, the intent/variant axis,
boolean naming, and the `asChild` holdouts to canonical form in a single major,
shipped with a migration guide and codemod. The phased alternative is retained in
§8 for context.

---

## 1. Methodology

The audit is data-driven, not impression-driven. The machine-readable component
catalog (`packages/ui/components.json`, generated by
[`scripts/generate-llms.ts`](../scripts/generate-llms.ts) from the real source
via `ts-morph`) was queried for every public prop, variant axis, and type
export. Findings were cross-checked against component source. Raw extraction
queries are reproducible from the catalog; representative ones appear inline.

**Severity rubric**

- **High** — affects the *core mental model* of the API (sizing, intent). A
  consumer cannot reliably predict the API of an unfamiliar component.
- **Medium** — local surprise. A consumer guesses wrong on a specific prop but
  recovers quickly from types/autocomplete.
- **Low** — ergonomic/DX gap; no wrong guesses, just missing affordances.

## 2. Finding 1 — Fragmented `size` vocabulary (High)

29 components expose a `size` axis, across **8 mutually incompatible scales**,
and — worse than the count — the *ordering and membership* disagree for the same
conceptual scale:

| Scale | # | Components |
| ----- | - | ---------- |
| `sm \| default \| lg` | 18 | Autocomplete, Checkbox, Input, Textarea, Slider, Tabs, Rating, … |
| `xs \| sm \| default \| lg \| xl` | 3 | Badge, Spinner, Toggle |
| `sm \| default \| lg \| xl \| full` | 2 | Alert Dialog, Dialog |
| `sm \| default` | 2 | Banner, Tooltip |
| `default \| xs \| sm \| lg \| xl \| icon \| icon-xs \| icon-sm \| icon-lg \| icon-xl` | 1 | Button |
| `default \| sm \| xs \| xl` | 1 | Item |
| `xs \| sm \| default \| lg` | 1 | Progress |
| `default \| sm \| lg` | 1 | Sidebar |

**The smell isn't variety — some components genuinely need more steps.** The
smell is that the *same steps are spelled in different orders* (`Item:
default|sm|xs|xl` vs `Spinner: xs|sm|default|lg|xl`) and that two components with
identical conceptual scales (`Progress: xs|sm|default|lg` vs the 18-member
`sm|default|lg` group) don't share one. There is no documented canonical scale
for a new component to copy.

**Impact.** A consumer who learns `size="sm"` on `Input` cannot assume `sm` is
the smallest step elsewhere (it isn't on Badge, Button, Spinner, Item). Theming
and layout code that switches on size can't be shared across components.

**Recommendation.** Define one **canonical, totally-ordered** scale in the
contract and require components to expose a *contiguous subset* of it, always in
canonical order:

```
xs · sm · default · lg · xl
```

with two documented, namespaced extensions used only where the primitive truly
needs them:

- **Overlay width** (`Dialog`, `Alert Dialog`): may add `full`.
- **Square controls** (`Button`): may add the `icon` / `icon-{step}` family,
  which is a *shape* modifier orthogonal to size and arguably belongs on its own
  `shape` axis (see Finding 2's theme — axis naming should describe the concept).

Converging the 18-member `sm|default|lg` group is free (it's already a subset).
The outliers (`Item`, `Progress`, `Sidebar`) reorder/rename to match.

## 3. Finding 2 — `variant` vs `intent` axis, with overlapping values (High)

The semantic axis is named **`variant`** on 20 components but **`intent`** on 4
(Avatar, Checkbox, Empty, Toggle). That alone is a coin-flip for consumers. The
deeper problem, visible once you read the actual values, is that **`intent` is
overloaded across three unrelated concepts**, and the *same* semantic-colour
values appear under *both* axis names:

```
intent  Avatar:   online | busy | away | offline                  ← presence STATUS (not colour!)
intent  Checkbox: default | destructive                           ← semantic colour (2-step)
intent  Empty:    muted | info | success | warning | destructive  ← semantic colour
intent  Toggle:   default | destructive | success | info | warning ← semantic colour

variant Alert:    default | info | success | warning | destructive | invert   ← semantic colour
variant Banner:   info | success | warning | destructive                      ← semantic colour
variant Card:     default | outline | ghost                                   ← fill/emphasis
variant Tabs:     default | line | underline                                  ← style
variant Badge:    default | outline | secondary | info | success | warning | destructive | invert
                  | primary-light | warning-light | … | destructive-outline | … | ghost | link   ← colour × fill, FLATTENED
variant Button:   default | secondary | outline | brand-solid | brand | brand-outline
                  | destructive-solid | destructive | destructive-outline | … | ghost | link      ← colour × fill, FLATTENED
```

Three distinct concepts are tangled across these two prop names:

1. **Intent** — *semantic colour meaning*: `info | success | warning |
   destructive | brand`. Appears under `intent` (Checkbox, Empty, Toggle) **and**
   `variant` (Alert, Banner) — same values, two prop names.
2. **Emphasis / fill** — `solid | soft | outline | ghost | link` (Button encodes
   this as the `-solid`/`-outline` suffix; Card/Item as `outline`/`ghost`).
3. **Presence status** — Avatar's `online | busy | away | offline` is genuinely a
   *different domain concept* mislabelled `intent`; it should be its own axis
   (e.g. `status`), not share a name with colour.

`Badge.variant` and `Button.variant` **flatten intent × emphasis into one
enum** (`brand`, `brand-solid`, `brand-outline`, `destructive-outline`,
`*-light`, …) — 24 values on Badge. That cartesian-product flattening is why
those lists are unreadable: two orthogonal axes crushed into one.

**Impact.** The single most-reached-for axis in the library has no predictable
name *or* shape. A consumer cannot transfer `<Badge variant="success">`
knowledge to `<Checkbox intent="success">` without checking docs, and cannot tell
whether `variant` will mean colour, shape, or fill until they read the type.

**Recommendation.** Standardise the axis *names by concept*, and stop flattening
orthogonal concepts:

- **`intent`** for semantic colour everywhere: `default | brand | success |
  warning | info | destructive`. Move Alert/Banner's colour values off `variant`
  onto `intent`.
- **`emphasis`** (or keep `variant` *only* for fill style) for `solid | soft |
  outline | ghost | link`.
- **`status`** for Avatar's `online | busy | away | offline` — it is not an
  intent.

Decompose `Badge`/`Button`'s flattened enum into `intent` × `emphasis`. This is
the highest-leverage change in the audit: it turns the most-used and most-tangled
axis into something guessable. (Note: this is also the most invasive — every
`<Button variant="brand-solid">` call site becomes `intent="brand"
emphasis="solid"`.)

> **Naming decision needed from the team:** `intent` vs `variant` as the colour
> axis name. `intent` is more precise and already in the contract; `variant` is
> more conventional in the shadcn/Radix ecosystem consumers may come from. I lean
> `intent` for colour + `variant`/`emphasis` for fill, but this is a deliberate
> divergence from ecosystem muscle-memory and worth a team call.

## 4. Finding 3 — Boolean prop naming drift (Medium)

The library *mostly* follows the React-idiomatic rule of **unprefixed, positively
phrased** booleans (`disabled`×15, `open`×4, `selected`, `required`×3,
`multiple`, `readOnly`). Three sub-patterns break it:

**(a) Inconsistent `is*` prefix.** The prefix appears on a minority and
sometimes *collides* with an unprefixed sibling:

```
isLoading (×3)   vs the idiomatic  loading
isRequired (×1)  vs               required (×3)   ← both exist in the library
isActive (×2)    vs               active
isDragging, isFetchingMore, isFetchingNextPage
```

`isRequired` coexisting with `required` is the clearest defect — two names for
one concept, in one library.

**(b) Mixed `show*` / `hide*` polarity.** Negative booleans default to `false`
meaning "shown", which inverts the reader's intuition and makes
double-negatives (`hideIcon={false}`) common:

```
positive: showLabel, showInput, showLegend, showTotal, showFooter, showDot, showLineNumbers, showOutsideDays …
negative: hideIcon, hideGridLines, hideXAxis, hideYAxis, hideClearAllButton, hidePlaceholderWhenSelected, hideOnCollapse
```

**(c) `error: boolean` (×6: Loading Indicator, Inline Suggestion Input, Input
OTP, Phone Input, Field).** A boolean named `error` reads like it should carry
*the error* (a message). It's not wrong, but `invalid` (mirroring the
`aria-invalid` the contract already standardises on) is less ambiguous and pairs
naturally with an `error?: ReactNode` message prop later.

**Impact.** Local, recoverable surprises — autocomplete rescues the consumer —
but they accumulate into "this library is a little inconsistent," which erodes
the trust the enforced contract works hard to build.

**Recommendation.** Adopt and lint three rules:

1. **No `is*` / `has*` prefixes** on boolean props. `isLoading → loading`,
   `isRequired → required`, `isActive → active`, `isDragging → dragging`.
2. **Positive polarity only.** `hideIcon → showIcon` (default `true`),
   `hideGridLines → showGridLines`, etc.
3. **`error: boolean → invalid: boolean`**, reserving `error` for a future
   message prop.

These are mechanical, individually small, and the highest *consumer-facing*
share of the audit.

## 5. Finding 4 — `asChild` usage vs `render` (✅ Resolved — 1 false positive, 1 legitimate boundary)

**Original finding (overstated):** "2 components (`credenza.tsx`, `tree.tsx`) use
`asChild` instead of `render`; migrate both."

**Correction after reading source:**

- **`tree.tsx` was a false positive.** The scan matched the substring inside
  `hasChildItems`, not a real `asChild` prop. The `\basChild\b` enforcement
  regex never flagged it; it was an erroneous grandfather entry, now removed.
- **`credenza.tsx` is a legitimate, permanent exemption.** Credenza is a
  responsive overlay that renders a Base UI `Dialog` on desktop and a **vaul**
  `Drawer` on mobile. The desktop branch already uses `render`; the mobile
  branch's `asChild` lands on `DrawerTrigger`/`DrawerClose`, which are thin
  re-exports of vaul's primitives — and vaul's polymorphism API *is* `asChild`,
  with no `render` equivalent. Forcing `render` here would mean wrapping vaul to
  translate `render → asChild`, adding indirection for zero consumer-facing gain.

So there is **no migration to do**. The `asChild` ban (§14) is enforced in
`check-conventions.mjs`, with `credenza.tsx` recorded as a documented permanent
exemption (vaul boundary), exactly like `code-block.tsx` is exempt from the
raw-colour rule for its always-dark terminal surface.

**Lesson (again):** confirm scan-derived findings against source. Two of the
five findings in this audit (this one and §6) were scan artifacts, not real work.

## 6. Finding 5 — `*Props` type export gaps (Low)

The contract's "named `*Props` type for every component" holds for 97/124.
**27 components don't export one**, including `badge`, `banner`, `spinner`,
`tag-group`, `metric-card`, `timeline`, `tracker`, `shell`, `chart`,
`copy-button`, and most of the `*-with-*` pattern family.

`direction` and `checkbox-tree` are legitimately exempt (provider re-export /
render-prop). The rest are not — consumers wrapping them can't name their prop
type.

**Impact.** DX only: consumers must `React.ComponentProps<typeof X>` instead of
importing a named type. No runtime or correctness effect.

**Recommendation.** Export a named `*Props` for each non-exempt component;
fold the exemptions (provider/render-prop) into `check-conventions.mjs` so the
rule becomes enforceable rather than aspirational.

## 7. What's already excellent (and should be protected)

Worth stating plainly, because it's the foundation the above builds on:

- **Polymorphism is effectively 100% on `render`.** The one `asChild` site
  (credenza's vaul branch, Finding 4) is a legitimate library-boundary exemption,
  now enforced — the pattern is universal across the library's own primitives.
- **Enforced visual contract passes clean:** focus ring (`ring-3`), disabled,
  `aria-invalid` width, semantic-token-only colours, `cn()` merging, `"use
  client"` placement, `ComponentProps` typing.
- **Generated, drift-checked docs.** `llms.txt` / `components.json` are produced
  from source and gated in CI, so this audit could be run *from data* rather than
  by hand — itself a sign of a healthy system.

The lesson: the parts that are **machine-enforced are perfect**, the parts that
are **only documented (or not) drift.** That's the whole thesis of the
recommendation.

## 8. Recommended approach

> **Team decision (2026-05-30):** proceed with the **full rename in one breaking
> release**, covering all four actionable areas — size scale, intent/variant axis,
> boolean naming, and polymorphism (`asChild` → `render`). The phased path below
> is retained for context, but the chosen execution is the single-release variant
> described at the end of this section.

**My recommendation: incremental, deprecation-aliased convergence.** Concretely,
a three-phase rollout:

**Phase 1 — Codify + freeze the bleed (no breaking changes).**
Write the canonical standards (size scale, `intent`/`emphasis`/`shape` axes,
boolean naming) into [`component-contract.md`](component-contract.md), and extend
`check-conventions.mjs` with rules that fail on **new** violations only
(grandfather the current set via the existing `EXEMPT` map). This stops the drift
from growing the moment it lands — the highest-ROI step, and fully non-breaking.

**Phase 2 — Migrate behind aliases (non-breaking, one or two minors).**
Rename props to canonical form while keeping the old name as a deprecated alias
(accept both; `@deprecated` JSDoc on the old one; warn in dev). Shrink the
`EXEMPT` grandfather set component-by-component as each is converged. Consumers
upgrade at their own pace; nothing breaks.

**Phase 3 — Drop aliases in one announced major.**
Once usage has moved, remove the deprecated aliases in a single, well-documented
breaking release with a codemod/migration guide.

**Why this over "rename everything now":** `@strait/ui` is `0.x` and gaining
adopters. A single hard-breaking pass is *cleaner in the repo* but pushes a
migration cost onto every consumer at once, for a change that is purely cosmetic
to them. The aliased path reaches the same end state with the disruption spread
out and opt-in. The one argument *for* doing it now is precisely that we're still
`0.x` — breaking changes are cheapest before 1.0. If the team's read is that the
adopter base is still small enough to absorb one bump, the "full rename now" path
is defensible; I'd still gate it behind a migration guide + codemod.

**Sequencing within scope:** do **Finding 2 (intent/variant)** and **Finding 1
(size)** first — they shape the core mental model and inform every new component.
**Finding 3 (booleans)** is the most mechanical and can run in parallel.
**Finding 4 (Props exports)** is a quick cleanup that can ride along with any of
them.

## 9. My opinion — is this a good improvement?

**Yes — but with honest framing about its tier.**

This is a **high-value, medium-cost** improvement, and it's the *right* one to do
**now** specifically because the library is pre-1.0. Consistency work gets
exponentially more expensive after you have a large installed base; doing it
while breaking changes are still cheap is exactly correct timing.

The value is real and compounding: a design system's ROI comes from *transfer* —
learning one component teaching you the next. Every naming inconsistency taxes
that transfer, and the tax is paid by every consumer on every component, forever.
Fixing it also makes the system easier for *us* to extend, and easier for LLM
consumers (who now read your `llms.txt`) to use correctly — a guessable API is a
guessable API for humans and models alike.

I'd set honest expectations on two things:

1. **It's invisible until it isn't.** This work won't demo well — no screenshot,
   no new feature. Its payoff is the *absence* of friction and a lower long-term
   maintenance slope. Frame it to stakeholders as paying down predictability debt
   while it's cheap, not as a feature.
2. **It's not the highest *user-facing* lever.** If the question is "what's the
   single best next improvement for end users," automated a11y enforcement
   (WCAG 2.2 AA per-story in CI) arguably ships more concrete user value, and the
   theming knobs (density/radius, preset themes) are more visible. This audit is
   the best **internal-quality / API-maturity** lever and the best *foundational*
   one. I'd rank it: do **Phase 1 of this audit now** (cheap, stops the bleed),
   then weigh a11y vs. theming for the next visible win.

Net: worth doing, worth doing soon, and worth doing the *incremental* way. The
fact that you already have a machine-enforced contract and generated catalog
means you're unusually well-positioned to execute it — the hard infrastructure
already exists; this is mostly extending it to the naming layer.
